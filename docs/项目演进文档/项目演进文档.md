# 项目演进文档 (Project Evolution Document)

## 📋 文档说明
本文档记录了基于流失博弈的社区微公交调度系统（Mobi-Churn）的开发及优化历程。按时间顺序总结了系统从最初的架构设计到性能优化、Bug 修复及训练环境稳定的关键节点。

---

## 🕒 时间轴与关键里程碑

### 1. 架构设计与基准建立 (2026-01-12)
*   **模型定义**：确立了以“乘客流失（Churn）”为核心信号的 GNN-RL 调度框架。
*   **双阶段仿真**：定义了 Stage 1 (Gym 极速训练) 与 Stage 2 (SUMO 真实性验证) 的架构。
*   **课程学习**：设计了 L0-L4 的阶梯式难度方案，并引入 $\rho$（有效负荷率）作为晋级指标。
*   **详见文档**：`流失驱动的社区微公交动态调度架构设计_合并版.md`

### 2. 深度训练性能优化 (2026-01-14)
针对原始模型在显存利用率（仅 800MB）和训练时长（25小时+）上的瓶颈进行了深度重构：
*   **模型层替换**：将自定义卷积替换为高性能的 `TransformerConv`，推理成本降低 80%。
*   **批量反向传播**：将逐样本 Q 值计算改为批量堆叠后一次性 backward，显存利用率显著提升且计算图更稳定。
*   **混合精度引入**：启用 AMP (Autocast) 进一步压缩显存并加速浮点运算。
*   **成果**：显存占用达到 20GB 级别，训练时间缩短至 10-16 小时，GPU 利用率提升至 80% 以上。
*   **详见文档**：`DQN_OPTIMIZATION_COMPLETE.md`

### 3. Structural Unserviceable 统计问题诊断 (2026-01-14)
*   **现象**：发现日志中 `structural_unserviceable` 数值（1715）异常偏高。
*   **诊断**：确认该数值并非 Bug，而是课程学习阶段订单池（324万单）的全局死单统计。由于当时代码会自动将 `max_requests` 设置为全量订单数，导致了该统计偏差。
*   **结论**：实际可达率仍保持在 99.9% 以上，不影响训练效果。
*   **详见文档**：`structural_unserviceable_diagnosis.md`

### 4. 训练稳定性与 NaN Loss 专项修复 (2026-01-15)
针对 L2/L3 阶段频繁出现的 `NaN Loss` 导致模型崩溃的问题进行了根本性修复：
*   **特征净化**：识别出图结构不连通产生的 `inf` 值是罪魁祸首。修改 `get_feature_batch`，对 ETA、乘客计数等特征进行硬性封顶 (`MAX_TIME_VAL = 36000.0`)。
*   **Loss 守卫**：在 DQN 优化器中添加 `torch.isnan(loss)` 检查，异常步自动跳过，确保权重不被污染。
*   **Bug 修复 (关键)**：发现并修复了课程学习中 `max_requests` 覆盖 Bug。现在配置文件中的 `1500` 单设置能正确生效，极大缓解了训练压力。

### 5. 训练数据分割与车队规模调整 (2026-01-15)
*   **时间划分**：实现 30% 训练集与 70% 评估集的时间轴分割（前18天用于训练，后43天用于评估）。
*   **运力匹配**：将车辆数从 50 调整至 51 辆（以 `max_requests: 1500` 为基准，确保系统运力盈余约 10%）。
